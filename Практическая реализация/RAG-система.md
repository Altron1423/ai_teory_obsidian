## 1. Что такое RAG и зачем он агенту
#определение 
**Retrieval-Augmented Generation (RAG)** — это архитектурный паттерн, при котором генеративная модель (LLM) при ответе на запрос дополняется релевантной информацией, извлечённой из внешнего источника знаний. Вместо того чтобы «помнить» всё внутри своих параметров, модель получает нужный контекст «на лету» через поиск.

Для агента RAG решает ключевые проблемы:
- **Долговременная память** — факты, которые не влезают в системный промпт или не должны быть статичными.
- **Актуальность** — доступ к свежим документам, логам действий, базе знаний.
- **Обоснованность** — снижение галлюцинаций, так как модель опирается на предоставленный текст.
- **Контролируемость** — вы управляете, какие знания доступны модели в конкретный момент.

---

## 2. Архитектура RAG: три этапа

1. **Индексирование (офлайн)**  
   Документы/данные разбиваются на фрагменты (чанки), для каждого вычисляется эмбеддинг, и всё сохраняется в векторную базу.

2. **Поиск (retrieval, онлайн)**  
   По запросу пользователя вычисляется эмбеддинг, и из базы выбираются наиболее семантически близкие чанки.

3. **Генерация (онлайн)**  
   Извлечённые чанки вставляются в промпт языковой модели, которая формирует финальный ответ, опираясь на них.

---

## 3. Подробно об индексировании

### 3.1. Чанкинг (нарезка документов)

Это критически важный шаг. От размера и способа нарезки зависит качество поиска.

- **Размер чанка**: зависит от embedding-модели (например, `nomic-embed-text` поддерживает до 8192 токенов, но оптимально 512–1024) и от характера данных. Маленькие чанки дают точность, но теряют контекст; большие — сохраняют контекст, но размывают смысл.
- **Перекрытие (overlap)**: соседние чанки пересекаются на несколько предложений (например, 10-20% длины). Это уменьшает риск, что важная мысль окажется разрезанной.
- **Методы нарезки**:
  - По фиксированному числу токенов (простой, но может резать посреди предложений).
  - По предложениям/абзацам с учётом семантических границ (лучше, требует NLP-инструментов типа spaCy или NLTK).
  - Иерархический чанкинг: документ делится на крупные секции, а те — на мелкие. При поиске можно использовать мелкие, а для контекста подтягивать родительскую секцию (родительский документ).

### 3.2. Выбор embedding-модели

В LM Studio вы загружаете отдельную модель для `/v1/embeddings`. Важные критерии:
- **Размерность вектора**: 384, 768, 1024... Чем больше, тем выше разрешающая способность, но медленнее поиск и больше памяти.
- **Максимальная длина токенов**: убедитесь, что модель переваривает ваши чанки без усечения.
- **Языковая поддержка**: multilingual-модели (`multilingual-e5`, `bge-m3`) предпочтительны, если знания на разных языках.
- **Качество**: можно ориентироваться на бенчмарк MTEB.

Для локального использования популярны:
- `nomic-embed-text-v1.5` (хороший баланс, до 8192 токенов)
- `bge-large-en` / `bge-small-ru`
- `text-embedding-3-small` (в формате GGUF)
- `all-MiniLM-L6-v2` (лёгкая, англ.)

### 3.3. Векторная база данных

После получения эмбеддингов их нужно где-то хранить и уметь быстро искать ближайших соседей.

- **Легковесные варианты**:  
  - **ChromaDB** — простая, на Python, хранит векторы и метаданные, удобна для прототипов.  
  - **FAISS** (Facebook) — высокопроизводительная библиотека C++ с обёртками для Python, работает в памяти.  
  - **Qdrant** — быстрая, с клиент-серверной архитектурой, фильтрацией по метаданным.  
- **Выбор для агента**: если всё работает локально и объём документов умеренный, ChromaDB или FAISS хороши. Если нужна фильтрация по датам/категориям — Qdrant.

Важно хранить вместе с вектором исходный текст чанка и метаданные (источник, дата, номер страницы и т.д.) — они пригодятся при формировании контекста.

## 4. Этап поиска (retrieval)

### 4.1. Базовый поток
1. Получаем запрос пользователя (можно предварительно очистить или расширить).
2. Вычисляем эмбеддинг запроса через `/v1/embeddings`.
3. Выполняем поиск top‑k ближайших соседей в векторной базе (обычно k = 3–10).
4. Возвращаем тексты чанков.

### 4.2. Продвинутые техники retrieval

- **Hybrid Search (гибридный поиск)**: комбинирует семантический (векторный) и лексический (BM25, TF-IDF) поиск. Это помогает, когда важны редкие термины (имена, коды), которые эмбеддинги могут упустить. Результаты объединяются через Reciprocal Rank Fusion.
- **Re-ranking (переранжирование)**: после грубого отбора топ‑N чанков (например, 20) применяется более точная, но медленная модель ранжирования (cross-encoder), чтобы оставить только самые релевантные 3–5. В локальном сценарии может быть накладно, но повышает точность.
- **Query Expansion (расширение запроса)**: LLM перефразирует или дополняет запрос пользователя, добавляя синонимы, уточнения, или генерирует гипотетический ответ (HyDE — Hypothetical Document Embeddings), чтобы улучшить поиск. Это требует дополнительного вызова LLM.
- **Self-query retrieval**: модель преобразует запрос в структурированный фильтр (например, «документы за 2023 год») плюс семантический вектор. Qdrant и ChromaDB поддерживают фильтрацию по метаданным.
- **Time-weighted retrieval**: более свежим документам придаётся больший вес (полезно для памяти агента, где недавние действия важнее старых).

### 4.3. Метрики сходства
- Косинусное сходство (стандарт для нормализованных векторов).
- Inner product (если векторы не нормализованы).
- Евклидово расстояние (реже).

LM Studio возвращает ненормализованные векторы, поэтому перед поиском стоит их нормализовать (если используете метрику косинуса). Многие библиотеки (FAISS) имеют встроенный индекс для косинусного сходства через inner product после нормализации.

## 5. Генерация ответа с контекстом
После того как получены чанки, их нужно вставить в промпт к `/v1/chat/completions`. Формат зависит от вашей архитектуры.
### 5.1. Куда вставлять
- **В `system` сообщение**: если контекст глобален для всей сессии (например, досье пользователя). Но тогда он не будет меняться от запроса к запросу, а значит, поиск нужно делать редко.
- **В `user` сообщение** (перед текущим запросом): классический вариант. Контекст прикладывается к каждому вопросу. Пример:
  ```
  Контекст:
  [чанк 1]
  [чанк 2]
  
  Вопрос: {user_query}
  ```
- **В отдельное `system`-сообщение непосредственно перед `user`**: это допустимо, модель увидит оба. Позволяет отделить «долговременные инструкции» от «ситуативного знания».

### 5.2. Цитирование источников
Чтобы модель ссылалась на источники, можно добавить в промпт инструкцию:
```
Отвечай, используя предоставленный контекст. Если ответа в контексте нет, скажи об этом. При цитировании указывай номер документа в квадратных скобках.
```
И разметить чанки номерами. Модель (особенно инструктивные) часто хорошо с этим справляется.

### 5.3. Сжатие контекста
Если чанков много, можно попросить LLM сжать их перед ответом (Map-Reduce: сначала суммаризовать каждый чанк, потом ответить) — но это дополнительные вызовы.

## 6. Интеграция RAG в вашего агента

Если у вас уже есть компоненты: системный промпт (характер, память), история диалога, недавние действия, картинки, вызовы инструментов. RAG должен органично вписаться.

### 6.1. Общая схема обработки запроса
1. Пользователь присылает текст (и/или картинку).
2. Если текст достаточно длинный/семантичный, вычисляете его эмбеддинг; если короткий (например «да», «продолжай»), можно пропустить поиск или использовать контекст из предыдущего запроса.
3. Выполняете retrieval из векторной БД (возможно, с гибридным поиском или фильтрацией по давности).
4. Собираете `messages`:
   - system: характер + долговременные правила + RAG-контекст (если постоянный) + сжатая история/память.
   - history (несколько последних user/assistant/tool сообщений).
   - user: текущий вопрос + прикреплённые картинки + оперативный RAG-контекст (если он динамический).
5. Отправляете в LLM, получаете ответ (и, возможно, tool_calls).
6. После ответа можно обновить память: записать новые факты в ту же векторную БД как отдельные чанки.

### 6.2. Память как векторная БД
Вы можете хранить «воспоминания» агента в той же коллекции, но с метаданными `type: memory` и `timestamp`. Тогда поиск будет возвращать и документы, и прошлые факты. Чтобы различать, можно использовать разные коллекции или фильтры.

### 6.3. Обновление и забывание
- **Добавление**: новые факты (например, «пользователь сказал, что боится пауков») эмбеддируются и складываются в БД.
- **Устаревание**: можно периодически удалять чанки старше определённого возраста или с низкой релевантностью.
- **Противоречия**: если найдены противоречивые факты, можно попросить LLM разрешить конфликт (но это сложно).

## 7. Продвинутые паттерны RAG

### 7.1. Agentic RAG
Агент сам решает, когда и как выполнять retrieval. Например:
- `tool_choice` с функцией `search_memory(query)` — модель вызывает «инструмент» для поиска, вы передаёте ей чанки в ответе `tool`, и она продолжает генерацию с ними. Это избавляет от необходимости всегда вставлять контекст и экономит место в промпте.
- Агент может выполнить несколько поисковых запросов, если первый не дал результатов.

В схеме с tool calling это особенно удобно: опишите функцию `retrieve_information(query: str)`, и пусть модель решает, нужно ли ей что-то искать.

### 7.2. Само-рефлексия (Self-RAG)
Модель генерирует специальные токены, оценивающие релевантность полученных чанков и полезность собственного ответа. Требует дообучения, для локального использования сложно, но идею можно имитировать через prompt engineering.

### 7.3. Multi-modal RAG
Поскольку модель поддерживает зрение, можно сохранять не только тексты, но и изображения. Подходы:
- Хранить описание изображения (сгенерированное заранее) в текстовом чанке, а само изображение — в отдельной папке, и при поиске подтягивать картинку и вставлять её в user-сообщение.
- Использовать мультимодальную embedding-модель (например, `nomic-embed-vision`), но в LM Studio это сложно — проще пока остановиться на текстовых описаниях.

## 8. Практические рекомендации для LM Studio

- **Embedding-модель загружайте отдельно**: либо через второй экземпляр LM Studio на другом порту, либо индексируйте базу один раз и сохраняйте векторы на диск (pickle/numpy), затем для поиска загружайте только LLM, а эмбеддинги запросов вычисляйте через ту же модель, переключая загрузку. Но это неудобно. Решение: использовать для эмбеддингов библиотеку `sentence-transformers` напрямую (она загрузит модель в процесс Python независимо от LM Studio) — тогда `/v1/embeddings` не нужен вообще, но вы получите полную совместимость.
- **Если используете `/v1/embeddings` с переключением моделей**: загрузили embedding-модель -> проиндексировали -> выгрузили, загрузили LLM -> работаете. При появлении новых документов придётся снова переключаться.
- **Оптимизация поиска**: FAISS на CPU может обрабатывать сотни тысяч векторов за миллисекунды, поэтому для локального агента проблем не будет.

## 9. Итоговая схема для вашего агента

```
Пользовательский ввод (текст + изображение)
       │
       ▼
┌─────────────────┐
│ Извлечение текста│ (если картинка - возможно, предварительный OCR/описание)
└────────┬────────┘
         ▼
┌─────────────────┐
│  Эмбеддинг запроса│  (через sentence-transformers или /v1/embeddings)
└────────┬────────┘
         ▼
┌─────────────────┐
│ Векторный поиск  │  (FAISS/ChromaDB) -> top-k чанков + возможно memory
└────────┬────────┘
         ▼
┌─────────────────┐
│ Сборка messages  │
│ - system: характер, правила, (опционально) RAG-контекст
│ - user: вопрос + картинки + контекст (если не в system)
│ - history: последние N сообщений
└────────┬────────┘
         ▼
┌─────────────────┐
│ /v1/chat/completions │ (с tools, если нужно)
└────────┬────────┘
         ▼
    Ответ ассистента
         │
         ▼
 (при необходимости) Сохранение важных фактов в векторную БД как новых чанков "памяти"
```

Эта архитектура позволяет гибко управлять знаниями агента, не засоряя системный промпт и не полагаясь только на статическую память. При желании retrieval можно обернуть в функцию-инструмент, чтобы модель сама решала, когда искать.
