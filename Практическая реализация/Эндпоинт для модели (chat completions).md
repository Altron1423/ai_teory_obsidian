
## 1. Общая информация

- **Полный URL** (в LM Studio): `http://localhost:1234/v1/chat/completions`
- **Метод**: `POST`  ```
- **Назначение**: создать ответ модели на основе списка сообщений с поддержкой системных инструкций, многоходовых диалогов, вызова инструментов и передачи изображений.

## 2. Параметры запроса (обязательные и важные)

### Обязательные
| Параметр   | Тип      | Описание                                                                                                        |
| ---------- | -------- | --------------------------------------------------------------------------------------------------------------- |
| `model`    | `string` | Идентификатор загруженной модели. Можно получить из `GET /v1/models` (в LM Studio это обычно имя файла или ID). |
| `messages` | `array`  | Список сообщений, определяющий контекст разговора. Формат каждого сообщения описан ниже.                        |
### Параметры управления генерацией (рекомендую всегда задавать явно)
| Параметр      | Тип                  | Описание                                                                                                                                                                                                                                                                                                                                                                                                                                                                    |
| ------------- | -------------------- | --------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| `max_tokens`  | `integer` или `null` | Максимальное количество токенов, которое модель может сгенерировать в ответе.<br>- Если `null` — используется значение по умолчанию из настроек сервера LM Studio.<br>- Слишком маленькое значение → ответ обрывается на полуслове (особенно критично при вызовах инструментов, где нужно закрыть XML-теги).<br>- Слишком большое — ответ может стать избыточным, модель может «болтать» дольше нужного.<br>*Для агентов с tool calling рекомендую ставить не меньше 2048.* |
| `temperature` | `number`, от 0 до 2  | Управляет «креативностью» / случайностью.<br>- Значения около 0 делают ответы более детерминированными и предсказуемыми (модель выбирает самые вероятные токены). Подходит для точных инструкций, вызова инструментов.<br>- Значения выше 0.7 увеличивают разнообразие, но могут приводить к уходу от темы и ошибкам в структуре вызовов.<br>*Типичный диапазон для агентов: 0.2–0.5.*                                                                                      |
| `top_p`       | `number`, от 0 до 1  | Nucleus sampling: модель рассматривает только токены, суммарная вероятность которых не превышает `top_p`.<br>- Альтернатива температуре. Можно использовать совместно, но обычно выбирают что-то одно.<br>- Низкий `top_p` (например, 0.1) делает ответы очень сфокусированными, высокий (0.9) — более разнообразными.<br>*Если задан, то обычно `temperature` поднимают до 1.0, но для агента чаще предпочитают чистую температуру без `top_p`.*                           |
| `stream`      | `boolean`            | Если `true`, ответ приходит в виде потока Server-Sent Events (чанки с `delta`).<br>- Позволяет показывать пользователю генерацию в реальном времени.<br>- При использовании инструментов стриминг требует аккуратной сборки аргументов из нескольких чанков. В LM Studio могут возникать ошибки парсинга tool call именно в потоковом режиме. Рекомендую начинать с `false` для отладки.                                                                                    |
| `stop`        | `string` или `array` | Последовательности, при появлении которых генерация немедленно прекращается.<br>- Полезно, если вы вручную формируете диалог в текстовом режиме (через `/v1/completions`), но в `/v1/chat/completions` применяется реже.<br>- Пример: `stop: ["\nПользователь:", "\nUser:"]` — модель остановится, не начав писать за пользователя.                                                                                                                                         |
| `seed`        | `integer`            | Фиксирует генератор случайных чисел.<br>- Обеспечивает воспроизводимость: одинаковые запросы с одинаковым `seed` дают одинаковый ответ (при прочих равных).<br>- Полезно для тестирования и отладки.                                                                                                                                                                                                                                                                        |
> LM Studio **не поддерживает** некоторые поля OpenAI: `logit_bias`, `response_format` (для structured output), `parallel_tool_calls`. Они просто игнорируются или вызывают ошибку. `n` (количество вариантов) также обычно не работает — LM Studio выдаёт только один ответ.
### Параметры для вызова инструментов
| Параметр      | Тип                   | Описание                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                  |
| ------------- | --------------------- | --------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| `tools`       | `array`               | список описаний функций, доступных модели. Каждый элемент — объект с полями:<br>- `type` (`"function"`)<br>- `function`: `name`, `description`, `parameters` (JSON Schema).  <br>Модель может решить вызвать одну из этих функций, если её ответ требует внешнего действия. Сами функции вы исполняете на стороне клиента.                                                                                                                                                                                                                                                                                                                                                                                |
| `tool_choice` | `string` или `object` | Управляет поведением модели в отношении вызовов инструментов:<br>- `"none"` — модель вообще не должна вызывать функции, даже если они переданы. Полезно, когда нужно временно запретить tool calling без удаления `tools` из запроса.  <br>- `"auto"` (по умолчанию) — модель сама решает, вызывать ли функцию и какую.<br>- `"required"` — модель обязана выполнить вызов. Если ей нечего вызвать, она может сгенерировать синтаксически неправильный вызов или ошибку (как у вас с незавершённым `<tool_call>`), поэтому используйте осторожно.<br>- Объект `{"type": "function", "function": {"name": "my_func"}}` — принудительный вызов конкретной функции. Модель сразу заполнит аргументы для неё. |
### Прочие параметры (меньшей важности)
| Параметр            | Тип                      | Описание                                                                                                                                                                                                                                                   |
| ------------------- | ------------------------ | ---------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| `frequency_penalty` | `number`, от -2.0 до 2.0 | Штрафует токены пропорционально тому, как часто они уже встречались в тексте. Положительные значения снижают вероятность повторений, отрицательные — наоборот, поощряют повтор. Для агентов редко требуется, но может помочь, если модель «зацикливается». |
| `presence_penalty`  | `number`, от -2.0 до 2.0 | Штрафует токены, которые хотя бы раз появлялись в тексте. Положительные значения стимулируют модель использовать новые слова и не повторяться.                                                                                                             |
| `user`              | `string`                 | Идентификатор конечного пользователя. В LM Studio не влияет на ответ, пересылается «как есть».                                                                                                                                                             |

## 3. Формат сообщений (`messages`)

Каждое сообщение – объект с полями `role` и `content`. `role` может быть:
- `"system"` – инструкция для модели (характер, правила, память, RAG-контекст).
- `"user"` – запрос пользователя (текст, картинка, или вместе).
- `"assistant"` – ответ модели (можно вставлять для поддержки истории).
- `"tool"` – результат вызова инструмента (идентификатор + содержание).

### Контент сообщения
- **Простой текст**: `"content": "строка"`.
- **Массив частей** (мультимодальный):
  ```json
  "content": [
    {"type": "text", "text": "описание"},
    {"type": "image_url", "image_url": {"url": "data:image/png;base64,...."}}
  ]
  ```
  Можно прикреплять несколько изображений, чередуя с текстом. Подробнее о картинках — в разделе 5.

### Сообщения `assistant`
Могут содержать не только текст, но и **`tool_calls`** — массив вызовов функций, которые модель хочет выполнить. Пример:
```json
{
  "role": "assistant",
  "content": null,
  "tool_calls": [
    {
      "id": "call_abc123",
      "type": "function",
      "function": {
        "name": "get_weather",
        "arguments": "{\"location\":\"Moscow\"}"
      }
    }
  ]
}
```
После выполнения нужно добавить сообщение с ролью `"tool"`:
```json
{"role": "tool", "tool_call_id": "call_abc123", "content": "25°C, солнечно"}
```
И затем продолжить диалог.

## 4. Ответ сервера

### Не стриминговый ответ
Пример тела ответа:
```json
{
  "id": "chatcmpl-...",
  "object": "chat.completion",
  "created": 1234567890,
  "model": "model-id",
  "choices": [
    {
      "index": 0,
      "message": {
        "role": "assistant",
        "content": "Ответ текстом...",
        "tool_calls": null  // или массив вызовов
      },
      "finish_reason": "stop"
    }
  ],
  "usage": {
    "prompt_tokens": 500,
    "completion_tokens": 150,
    "total_tokens": 650
  }
}
```
- `finish_reason` может быть: `"stop"` (естественное завершение), `"length"` (достигнут `max_tokens`), `"tool_calls"` (модель хочет вызвать функцию), `"content_filter"` (редко).

### Стриминг (`stream: true`)
Ответ приходит через SSE с частями. Каждый чанк имеет:
```json
{
  "id": "chatcmpl-...",
  "object": "chat.completion.chunk",
  "choices": [
    {
      "index": 0,
      "delta": {
        "role": "assistant",
        "content": "фрагмент текста",
        "tool_calls": [{"index": 0, "function": {"name": "get_weather", "arguments": "{\""}}]
      },
      "finish_reason": null
    }
  ]
}
```
Последний чанк содержит `finish_reason`. Для инструментов аргументы приходят частями, их нужно склеивать.

> В LM Studio: при стриминге ошибки парсинга tool calls могут возникать чаще. Если есть проблемы, попробуйте временно отключить стриминг.

## 5. Работа с изображениями (Vision)

Некоторые модели (например Qwen3.5-9B) поддерживают зрение, поэтому им можно передать картинки. Формат:
```json
{
  "role": "user",
  "content": [
    {"type": "text", "text": "Что на этой картинке?"},
    {
      "type": "image_url",
      "image_url": {
        "url": "data:image/jpeg;base64,<base64-строка>"
      }
    }
  ]
}
```
- **`url`** – только base64, ссылки на внешние ресурсы LM Studio не загружает.
- Параметр `detail` (как в OpenAI) LM Studio обычно игнорирует, изображение обрабатывается в максимальном качестве, которое позволяет модель.
- Можно передавать несколько картинок в одном сообщении, чередуя текст и изображения. Модель «видит» их по порядку.
- Картинки занимают много токенов контекста, поэтому длинная история с изображениями быстро расходует окно. Старые картинки рекомендуется удалять из истории.

## 6. Инструменты (Tool calling) в деталях
### a) Описывайте функции максимально чётко
Пример:
```python
tools = [
    {
        "type": "function",
        "function": {
            "name": "console_print",
            "description": "Выводит сообщение в консоль агента",
            "parameters": {
                "type": "object",
                "properties": {
                    "text": {"type": "string", "description": "Текст для вывода"}
                },
                "required": ["text"]
            }
        }
    }
]
```

### b) Управляйте вызовом через `tool_choice`
- `"auto"` – LM Studio позволит модели вызывать функции по необходимости. Это режим по умолчанию.
- `"required"` – модель обязана сделать вызов (иначе может ответить текстом? По спецификации она должна вызвать, но иногда отвечает текстом с ошибкой, как у вас).
- `{"type": "function", "function": {"name": "my_func"}}` – принудительно вызвать конкретную функцию.

Если вы уверены, что на определённом шаге нужен вызов, используйте `"required"` или укажите имя функции. Это уменьшит вероятность того, что модель начнёт болтать текстом и оборвёт вызов.

### c) Избегайте обрыва вызова
- Увеличьте `max_tokens` (минимум до 1024–2048).
- Добавьте в `system` явное указание: «Всегда закрывай XML-теги вызовов».
- Проверяйте, нет ли в истории сообщений незакрытых или «испорченных» тегов. Если модель ранее сгенерировала текст, похожий на `<tool_call>`, она может запутаться.

### d) Обработка цикличных ретраев
Ваш клиент, вероятно, при получении ошибки от LM Studio повторяет запрос с теми же параметрами. LM Studio снова падает при парсинге, и так по кругу. **Решение**: перехватывайте ошибку парсинга (код 400/500, сообщение «Failed to parse tool call») и НЕ повторяйте тот же запрос. Вместо этого можно:
- Попробовать отправить запрос с `tool_choice="none"` и попросить модель ответить текстом.
- Удалить последнее сообщение пользователя, которое вызвало сбой, и предложить уточнённый запрос.
- Логировать и переходить к fallback-действию.

## 7. Управление контекстом для агента

Ваша цель — сформировать системный промпт, содержащий характер, память, RAG-контекст и историю. Я рекомендую такую структуру:

- **Одно `system` сообщение**: в него поместите всё постоянное и долговременное (характер, инструкции, правила, RAG-факты, если они не меняются).
- **Динамический RAG**: если результаты поиска меняются каждый запрос, добавляйте их в начало сообщения `user` перед вопросом (например, «Релевантная информация: ... \n Вопрос: ...»). Это привлечёт внимание модели к контексту именно сейчас.
- **История диалога**: сообщения `user` и `assistant` накапливайте в массиве. Старые записи удаляйте, когда приближаетесь к лимиту контекста (у LM Studio есть настройка «Context Length» на сервере, или можно задать при загрузке модели через `/api/v1/models/load`). Удаляйте и старые изображения — они самые «тяжёлые».
- **Системный промпт можно сделать огромным** — это удобно хранить в одном месте. Но проверьте, что модель правильно воспринимает длинные инструкции (Qwen3.5 умеет).

## 8. Особенности LM Studio

- **Автоматический чат-шаблон**: При использовании instruct-моделей LM Studio вставляет специальные токены форматирования. Вы не должны беспокоиться об этом, если работаете через chat endpoint — он сам подставит `system`, `user`, `assistant` в соответствии с шаблоном модели.
- **Лимиты**: Размер контекста задаётся при загрузке модели. Максимальное число токенов генерации (`max_tokens`) не может превышать оставшееся место в окне.
- **Ошибки валидации**: LM Studio более строга к структуре сообщений, чем может показаться. Например, в ответе `assistant` не может быть и `content` и `tool_calls` одновременно (хотя OpenAI разрешает). Придерживайтесь стандарта: если есть tool_calls, `content` должен быть `null` (или вообще отсутствовать). Если модель всё же выдала и то и другое, LM Studio отбросит вызов или упадёт.
- **Логи**: в консоли LM Studio видно, когда парсинг вызова не удался. Это помогает отладить.

## 9. Вариант итогового шаблона запроса для агента

```python
response = client.chat.completions.create(
    model="qwen3.5-9b",  # актуальный ID
    messages=[
        {
            "role": "system",
            "content": (
                "Ты — саркастичный дворецкий. Твоя память: ...\n"
                "Правила: ...\n"
                "RAG-контекст: ..."
            )
        },
        {
            "role": "user",
            "content": [
                {"type": "text", "text": "Вот результаты поиска по базе: ...\nВопрос: опиши это фото"},
                {"type": "image_url", "image_url": {"url": f"data:image/png;base64,{base64_image}"}}
            ]
        }
    ],
    tools=[...],
    tool_choice="auto",  # или "required"
    max_tokens=2048,
    temperature=0.3,
    stream=False  # начните с False для стабильности
)
```
