## 1. Назначение

**`POST /v1/embeddings`** превращает текст в векторное представление (эмбеддинг) — массив чисел фиксированной длины, который отражает смысл текста. Эти векторы можно сравнивать (например, косинусным сходством), чтобы находить семантически близкие фрагменты.  
В контексте агента это основа для **RAG**:

- Вы заранее разбиваете документы/память на куски и получаете их эмбеддинги.
- При запросе пользователя вычисляете эмбеддинг запроса и ищете ближайшие куски.
- Найденный текст вставляете в `system` или `user` сообщение к `/v1/chat/completions`.

---

## 2. Параметры запроса

### Обязательные поля:
| Параметр | Тип                  | Описание                                                                                                                                          |
| -------- | -------------------- | ------------------------------------------------------------------------------------------------------------------------------------------------- |
| `model`  | `string`             | Идентификатор модели для эмбеддингов. В LM Studio это должна быть **специальная embedding-модель**, загруженная через интерфейс.                  |
| `input`  | `string` или `array` | Текст(ы) для векторизации. Можно передать одну строку или массив до 2048 строк (в OpenAI; LM Studio обычно поддерживает батчи разумного размера). |
### Опциональные (поддержка LM Studio может отличаться):
| Параметр          | Тип       | Назначение                                                                                                                 |
| ----------------- | --------- | -------------------------------------------------------------------------------------------------------------------------- |
| `encoding_format` | `string`  | `"float"` (по умолчанию) или `"base64"`. LM Studio обычно отдаёт только `float`.                                           |
| `dimensions`      | `integer` | Желаемая размерность вектора. **В LM Studio не поддерживается** — модель всегда возвращает вектор своей стандартной длины. |
| `user`            | `string`  | Идентификатор пользователя. Игнорируется.                                                                                  |
### Пример запроса (JSON):
```json
{
  "model": "text-embedding-nomic-embed-text-v1.5",
  "input": ["Как приготовить чай?", "Рецепт борща"]
}
```

## 3. Формат ответа

Успешный ответ (HTTP 200):
```json
{
  "object": "list",
  "data": [
    {
      "object": "embedding",
      "index": 0,
      "embedding": [0.0123, -0.0456, ...]
    },
    {
      "object": "embedding",
      "index": 1,
      "embedding": [0.0789, 0.0321, ...]
    }
  ],
  "model": "text-embedding-nomic-embed-text-v1.5",
  "usage": {
    "prompt_tokens": 15,
    "total_tokens": 15
  }
}
```
- `embedding` — массив чисел с плавающей точкой. Длина зависит от модели (обычно 384, 768, 1024 и т.д.).
- `index` соответствует порядку строк в `input`.
- `usage` показывает количество токенов в запросе (помогает оценить стоимость, но в локальной среде не критично).

## 4. Особенности в LM Studio

### a) Не любая модель подходит
`/v1/embeddings` требует **специализированную embedding-модель**, а не обычную чат-модель.  
Примеры популярных моделей, которые можно скачать в LM Studio:  
- `nomic-embed-text` (серия)  
- `bge-large-en` / `bge-small-ru`  
- `text-embedding-3-small` (в формате GGUF)  
- `multilingual-e5` и другие.

Если загружена только чат-модель (например, Qwen3.5-9B), то запрос к `/v1/embeddings` с её именем, скорее всего, **завершится ошибкой** `"Model not found or not an embedding model"`. Нужно сначала скачать и загрузить отдельную embedding-модель (в LM Studio можно быстро сменить модель через GUI или `/api/v1/models/load`, но помните, что одновременно активна только одна модель, поэтому придётся переключаться между LLM и эмбеддером — либо держать два отдельных сервера/порта).

### b) Параметр `dimensions` не работает
Даже если вы укажете `dimensions`, LM Studio его проигнорирует — вы всегда получите эмбеддинг полной размерности, заложенной в модели.

### c) Максимальная длина текста
У каждой embedding-модели есть свой лимит токенов (например, 512 токенов для многих). Если вы подадите слишком длинный текст, LM Studio может либо обрезать его молча, либо вернуть ошибку. Рекомендуется заранее разбивать документы на чанки, которые гарантированно укладываются в лимит.

### d) Батчинг
Можно отправить массив из нескольких строк — это эффективнее, чем делать отдельные запросы на каждую. LM Studio обработает их параллельно.

### e) Кэширование и производительность
Эмбеддинги вычисляются на CPU/GPU в зависимости от настроек. Если вам нужно проиндексировать большую базу знаний, загружайте embedding-модель с GPU-ускорением, чтобы ускорить процесс.

## 5. Пример вызова через OpenAI-клиент (Python)

```python
from openai import OpenAI

client = OpenAI(
    base_url="http://localhost:1234/v1",
    api_key="lm-studio"
)

response = client.embeddings.create(
    model="text-embedding-nomic-embed-text-v1.5",  # ваша загруженная модель
    input=["Ваш текст для векторизации"]
)

embedding = response.data[0].embedding
print(len(embedding))  # размерность вектора
```

Если нужно получить эмбеддинг для нескольких текстов разом:
```python
texts = ["факт 1", "факт 2", "факт 3"]
response = client.embeddings.create(model=..., input=texts)
embeddings = [item.embedding for item in response.data]
```

## 6. Интеграция в [[RAG-система|RAG-систему]]

### Шаг 1: Индексирование базы знаний
- Разбейте документы на небольшие куски (чанки) длиной ~256–512 токенов.
- Для каждого чанка получите эмбеддинг через `/v1/embeddings`.
- Сохраните пары `(embedding, текст_чанка)` в векторную базу данных (например, Chroma, FAISS, Qdrant или даже простой numpy-массив).

### Шаг 2: Поиск по запросу пользователя
- Получите эмбеддинг запроса.
- Вычислите косинусное сходство с сохранёнными эмбеддингами (или используйте встроенный метод векторной БД).
- Выберите top‑k (например, 3–5) наиболее похожих чанков.

### Шаг 3: Формирование контекста
- Склейте текст выбранных чанков в строку.
- Вставьте эту строку в `messages` вашего `/v1/chat/completions`:
  - Либо в начало `system` (если факты должны действовать постоянно),
  - Либо в `user` перед вопросом (если контекст нужен только на этот запрос).

Пример:
```python
context = "\n".join(retrieved_chunks)
messages = [
    {"role": "system", "content": "Ты полезный ассистент."},
    {"role": "user", "content": f"Контекст:\n{context}\n\nВопрос: {user_query}"}
]
```

## 7. Важные советы

- **Держите embedding-модель отдельно.** Не пытайтесь загружать её одновременно с LLM в одном экземпляре LM Studio, если не хотите постоянно переключаться. Выходы: использовать два порта (запустить две LM Studio, или один сервер с LLM, а для эмбеддингов использовать библиотеку `sentence-transformers` напрямую, но тогда это уже не `/v1/embeddings`). Однако если база знаний индексируется разово, можно загрузить embedding-модель, построить индексы, а потом загрузить LLM — это допустимо.
- **Тестируйте размер чанка.** Слишком маленькие фрагменты теряют контекст, слишком большие — превышают лимит модели или размывают поиск. Подберите опытным путём.
- **Кэшируйте эмбеддинги.** Если база знаний не меняется, сохраните эмбеддинги на диск, чтобы не пересчитывать при каждом запуске агента.
- **Можно использовать несколько embedding-моделей.** Например, одну для русского, другую для английского — просто передавайте соответствующее `model` в запросе.

Таким образом, `/v1/embeddings` — ваш главный инструмент для превращения неструктурированных текстов в математические векторы, которые позволяют агенту «вспоминать» релевантную информацию и встраивать её в промпт к основной модели.
